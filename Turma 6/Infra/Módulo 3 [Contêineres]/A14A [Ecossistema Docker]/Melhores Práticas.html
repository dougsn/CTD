Usar imagens oficiais

No Docker Hub, encontraremos 2 tipos de imagens. As oficiais, publicadas por organizações (por exemplo, Ubuntu) ou aquelas publicadas por usuários individuais. Como diferenciá-las?

As imagens oficiais são marcadas como tal no site do Docker Hub. Se procurarmos pelo Ubuntu, veremos que as imagens publicadas pela Canonical são rotuladas como Docker Official Image.

No entanto, a maneira mais prática de entender se uma imagem é oficial ou não é o nome. As imagens não oficiais incluem o nome do usuário que as postou, enquanto as oficiais não. Vamos ver a diferença ao executar "docker pull". Para uma imagem oficial seria: docker pull ubuntu:latest. Enquanto para uma imagem que não é, o comando ficaria assim: docker pull user/nginx:latest.

No caso da imagem não oficial, provavelmente contém um servidor Nginx, mas não sabemos (pelo menos sem uma análise mais aprofundada) quais outras modificações o usuário que postou a imagem pode ter incluído.

Usar o comando COPY em vez de ADD

Ambos permitem adicionar ou copiar um arquivo de uma fonte externa para uma imagem do Docker. ADD foi o primeiro a ser criado para a especificação Dockerfiles. A grande diferença é que o ADD suporta a cópia de diferentes fontes. Por exemplo, você pode adicionar um arquivo hospedado na web, pode adicionar um arquivo que está em um sistema de arquivos da máquina que está compilando o Dockerfile e também pode descompactar um arquivo .tar (tarball, um formato de empacotamento nativo do Unix/Linux). O problema é que ADD – fazendo muitas coisas – pode levar à confusão, incluindo alguns problemas técnicos (e se eu quiser copiar um arquivo .tar para a imagem em vez de descompactá-lo?) e até mesmo problemas de desempenho.

COPY, por outro lado, foi introduzido posteriormente para cumprir a função específica de copiar um arquivo de um sistema de arquivos local para a imagem. E o COPY inclusive, tendo um propósito mais específico, torna o Dockerfile mais fácil de ler.

Em linhas gerais, recomenda-se:

  • Usar COPY para copiar arquivos locais para a imagem no momento da compilação.
  • Usar RUN para executar curl e atrelar com outros comandos para baixar um arquivo de uma fonte da Web e copiá-lo, instalá-lo ou descompactá-lo.

  • Usar ADD quando tiver um arquivo .tar e quiser descompactá-lo na imagem que está construindo.

Usar multi-stage builds

Essa é uma prática um tanto complexa, mas vale a pena saber que ela existe.

Ela nos permite gerar imagens menores incluindo explicitamente apenas os componentes necessários para executar um aplicativo.

Até agora, você deve ter visto Dockerfiles que incluem um único FROM. Em compilações de vários estágios, os Dockerfiles têm duas ou mais ocorrências de FROM. Isso produz várias imagens etéreas durante o processo de "compilação" e nos permite copiar elementos entre elas, até gerar uma imagem final apenas com o estritamente necessário para executar a aplicação.

Nesta prática, em geral, é utilizada uma primeira imagem base (especificada no primeiro FROM), na qual copiamos o código fonte da nossa aplicação e temos as ferramentas necessárias para compilá-la.

Em seguida, indicamos um segundo FROM dentro do mesmo Dockerfile, referenciando uma imagem base que possui apenas o

necessário para executar o aplicativo, mas não as ferramentas de compilação. E copiamos o produto gerado da compilação do primeiro container para o segundo.

Não gerar dependências externas

Os containeres que construímos precisam ser portáteis. Não importa onde eles sejam executados, eles devem ter tudo o que precisam para executar os aplicativos que residem neles.

É natural que os desenvolvedores sejam tentados a montar diretórios na máquina em que estão desenvolvendo em um container para acelerar o ciclo de desenvolvimento. Em vez de ter que construir a imagem toda vez que você quiser testar seu aplicativo e copiar os artefatos necessários para ele, é mais fácil mapear um diretório local e deixar os componentes atualizados lá.

Essa prática pode levar a dependências não resolvidas ao executar aplicativos em produção.

Portanto, ao contrário do que está especificado aqui:https://docs.docker.com/develop/dev-best-practices/#differences-in-development-and-production-environments, recomendamos que você não use volumes de tipo bind em ambientes de desenvolvimento.

Concatenar comandos

Muitas vezes encontraremos linhas como esta em Dockerfiles:

RUN apt-get update && apt-get upgrade -y

Em vez de algo como:

RUN apt-get update

RUN apt-get upgrade -y

Por quê? Cada linha em um Dockerfile produz uma nova camada em nossa imagem. Portanto, executar esses dois comandos separadamente produziria duas camadas. Nesse caso, ambos os comandos têm a mesma finalidade: atualizar os pacotes instalados no sistema operacional. Portanto, faz sentido concatená-los, pois estão intimamente relacionados, e executar um sem executar o outro não faria sentido. Dessa forma, otimizamos o uso de recursos e o tempo de compilação.

Você pode encontrar mais informações sobre essa prática em:https://docs.docker.com/develop/dev-best-practices/#how-to-keep-your-images-small.
